### Migration 18: 対応歴ラベルリネーム

```sql
UPDATE public.object_definitions
   SET label = '対応歴', updated_at = now()
 WHERE id = 'activities' AND label = '活動履歴';

UPDATE public.field_definitions
   SET label = '対応歴ID', updated_at = now()
 WHERE object_id = 'activities' AND field_name = 'id' AND label = '活動ID';
```

### Migration 19: duration_minutes 削除 + MV 再作成
**ファイル**: `supabase/migrations/19_drop_activities_duration.sql`

⚠️ **重要**: `mv_monthly_activities` がカラムに依存しているため、必ず1トランザクションで実行すること。
```sql
BEGIN;
DROP MATERIALIZED VIEW IF EXISTS public.mv_monthly_activities;
ALTER TABLE public.activities DROP COLUMN IF EXISTS duration_minutes;
DELETE FROM public.field_definitions
 WHERE object_id = 'activities' AND field_name = 'duration_minutes';
UPDATE public.reports
   SET definition = jsonb_set(
         definition,
         '{columns}',
         (SELECT COALESCE(jsonb_agg(col), '[]'::jsonb)
          FROM jsonb_array_elements(definition->'columns') AS col
          WHERE col->>'source' <> 'act.duration_minutes')
       )
 WHERE definition->'columns' @> '[{"source":"act.duration_minutes"}]';
CREATE MATERIALIZED VIEW public.mv_monthly_activities AS
SELECT
  owner_id,
  date_trunc('month', registered_datetime)::date AS month,
  d_bunrui,
  m_bunrui,
  COUNT(*) AS activity_count,
  COUNT(DISTINCT member_id) AS unique_member_count
FROM public.activities
WHERE deleted_at IS NULL AND registered_datetime IS NOT NULL
GROUP BY owner_id, date_trunc('month', registered_datetime), d_bunrui, m_bunrui;
CREATE UNIQUE INDEX idx_mv_mact_unique ON public.mv_monthly_activities(owner_id, month, d_bunrui, m_bunrui);
CREATE INDEX idx_mv_mact_month ON public.mv_monthly_activities(month);
CREATE INDEX idx_mv_mact_owner ON public.mv_monthly_activities(owner_id);
COMMIT;
```

## 7. 既知の問題と解決策（参考）

### 7.1 node:crypto が Edge/Serverless で動かない
- `createHash` → 独自 FNV ハッシュ (`hashKey()` in `activities_map.ts`)
- `randomUUID` → `globalThis.crypto.randomUUID()`
- **絶対に `node:crypto` を import しないこと**

### 7.2 RLS 「new row violates row-level security policy」エラー
- admin 操作（ユーザー削除・権限変更・パスワード設定）は `createServiceRoleClient()` を使う
- アプリ側で `getCurrentUser()` により admin であることを確認済みの前提で service role を使う

### 7.3 MV と ALTER TABLE の依存エラー
- `mv_monthly_activities` が `duration_minutes` に依存していたため `DROP COLUMN` が失敗
- 解決: `DROP MV → DROP COLUMN → CREATE MV` を1トランザクションで実行（migration 19）

### 7.4 Server Action の bodySizeLimit
- デフォルト 4MB の上限を `next.config.ts` で `25mb` に引き上げ済み
- それでも大きすぎる場合（18MB超）はスクリプトを使うよう UI でガイド

### 7.5 extract.csv の行数について
- `wc -l` で 1,211,736 と表示されるが、実データは **522,798 行**
- CSV の中にクォートされた改行（Description フィールドの改行）が含まれるため

## 10. セキュリティ上の絶対ルール

1. `SUPABASE_SERVICE_ROLE_KEY` はサーバー側のみ。クライアント側への露出禁止
2. 個人情報を含むレスポンスは RLS で強制制御
3. ログに個人情報を出さない（マスキング）
4. 物理削除禁止 → `deleted_at timestamptz` で論理削除
5. 動的 SQL でのユーザー入力の文字列連結禁止（ホワイトリスト + パラメータ化必須）
